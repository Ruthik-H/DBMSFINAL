🗄️ Phase 2 — Database Design
Key Concepts You Learned:

Concept   	What It Means
Primary Key	Uniquely identifies every row
Foreign Key	References another table's primary key
Normalization	Every piece of data lives in exactly ONE place
1NF	        Columns are atomic, no duplicates
2NF	        No partial dependencies
3NF	        No transitive dependencies
Junction/Bridge Table	Connects two tables with foreign keys
Data Redundancy	Storing same data twice — always avoid!


create a table structure

🔐 Phase 3 — Authentication Design
Key Concepts You Learned:

Concept	                What It Means
REST	                Rules for how frontend and backend communicate via HTTP
JWT Token	        Like a movie ticket — proves who you are without server storing anything
Stateless	        Server remembers nothing — JWT carries the identity
RBAC	                Role Based Access Control — each role sees only their content
Password Hashing	Never store raw passwords — always hash!
Authorization	        What you can DO after you're logged in
Authentication	        Proving WHO you are
SECRET_KEY	        Wax seal — signs and verifies token integrity. Never changes!
ALGORITHM	        HS256 — SHA256 hashing to sign tokens
Token Expiry	        Token dies after 30 mins — user must login again



JWT FLOW:

1. User logs in with email + password
        ↓
2. Server verifies against database
        ↓
3. Server creates JWT with (user_id, role, expiry)
        ↓
4. Token sent to frontend
        ↓
5. Every request → frontend sends token
        ↓
6. Server reads and verifies token
        ↓
7. No database lookup needed for identity!


SECRET_KEY = Stamp (stays on server FOREVER)
JWT Token  = Stamped Letter (travels to frontend)
